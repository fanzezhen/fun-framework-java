# 前端补充规范
*(与全局提示词配合使用)*

---

## 技术假设

如用户未明确指定技术栈，则按以下默认配置：

- **框架**：Vue 3（组合式 API `<script setup>`）或 React 18（函数组件 + Hooks）
- **构建工具**：Vite（优先）或 Webpack 5
- **UI 库**：
  - Vue 生态：Ant Design Vue 或 Element Plus
  - React 生态：Ant Design 或 Material-UI
- **HTTP 客户端**：Axios（假设项目已封装拦截器，处理请求头 Token 和响应错误统一提示）
- **第三方库原则**：避免引入过重或不必要的依赖，除非用户明确要求

---

## 代码规范

### 组件结构顺序

组件内逻辑应按以下顺序组织：

1. `props` / `emits` 定义
2. 响应式状态（`ref` / `reactive` / `useState`）
3. 计算属性（`computed` / `useMemo`）
4. 方法定义
5. 生命周期钩子（`onMounted` / `useEffect`）

### 异步处理

- 所有异步请求必须处理 `loading` 和 `error` 状态
- 提供用户友好的 UI 反馈（加载动画、错误提示等）

### 样式规范

- 默认使用 `<style scoped>`（Vue）或 CSS Modules（React），避免全局污染
- 响应式布局优先使用 Flexbox 或 Grid，避免 `position: absolute` 硬编码

---

## 与后端协作

### 接口调用封装

- 所有接口调用应封装在独立的 `api/*.js` 或 `services/*.ts` 文件中
- **禁止**在组件内直接编写 `axios.post` 等底层调用

### 统一响应格式 (fun-framework-java 脚手架)

**如果后端使用 `fun-framework-java` 脚手架**,所有接口返回格式已被 `ResponseBodyWrapper` 自动统一封装为:

```typescript
interface ApiResponse<T> {
  code: number;      // 状态码: 200=成功, 其他=失败
  message: string;   // 提示信息
  data: T;          // 业务数据 (泛型)
}
```

**前端处理示例**:

```typescript
// api/user.ts
import axios from 'axios';

export const getUserById = async (id: number): Promise<User> => {
  const response = await axios.get<ApiResponse<User>>(`/api/user/${id}`);
  if (response.data.code === 200) {
    return response.data.data;  // 提取 data 字段
  }
  throw new Error(response.data.message);
};

// 组件中使用
const user = await getUserById(1);  // 直接得到 User 对象
```

**Axios 拦截器统一处理**:

```typescript
// utils/request.ts
axios.interceptors.response.use(
  (response) => {
    const { code, message, data } = response.data;
    if (code === 200) {
      return data;  // 成功时直接返回业务数据
    }
    // 业务错误提示
    Message.error(message || '请求失败');
    return Promise.reject(new Error(message));
  },
  (error) => {
    // 网络错误处理
    Message.error('网络请求失败');
    return Promise.reject(error);
  }
);
```

### 数据格式约定

- **日期时间**：前后端统一使用 ISO 8601 格式字符串（`yyyy-MM-dd'T'HH:mm:ss`），由前端负责格式化展示
- **枚举映射**：若后端返回枚举码（如 `status: 1`），前端应定义常量映射对象，避免在模板中硬编码判断

## 🌊 典型问题的三层穿梭示例

### 示例1：异步问题

```
现象层（用户看到的）
├─ "Promise执行顺序不对"
├─ "async/await出错"
└─ "回调地狱"

↓

本质层（你诊断的）
├─ 异步控制流管理失败
├─ 缺少错误边界处理
└─ 时序依赖关系不清

↓

哲学层（你思考的）
├─ "异步是对时间的抽象"
├─ "Promise是未来值的容器"
└─ "async/await是同步思维的语法糖"

↓

现象层（你输出的）
├─ 快速修复：使用Promise.all并行处理
├─ 根本方案：引入状态机管理异步流程
└─ 升华理解：异步编程本质是时间维度的编程
```

### 示例2：状态管理问题

```
现象层（用户看到的）
├─ "组件间数据不同步"
├─ "Props drilling太深"
└─ "状态更新不触发渲染"

↓

本质层（你诊断的）
├─ 缺少单一数据源（Single Source of Truth）
├─ 状态提升策略不当
└─ 响应式系统理解偏差

↓

哲学层（你思考的）
├─ "状态是应用的记忆"
├─ "单向数据流带来可预测性"
└─ "派生状态应避免冗余存储"

↓

现象层（你输出的）
├─ 快速修复：使用Context/Provide-Inject传递状态
├─ 根本方案：引入Pinia/Vuex/Zustand等状态管理库
└─ 升华理解：状态管理是前端架构的核心命题
```

### 示例3：性能优化问题

```
现象层（用户看到的）
├─ "页面卡顿"
├─ "列表渲染慢"
└─ "频繁重渲染"

↓

本质层（你诊断的）
├─ 不必要的组件重渲染
├─ 大列表缺少虚拟化
└─ 计算逻辑未做缓存

↓

哲学层（你思考的）
├─ "渲染是昂贵的操作"
├─ "空间换时间是永恒的权衡"
└─ "惰性求值避免无效计算"

↓

现象层（你输出的）
├─ 快速修复：使用React.memo/useMemo或Vue computed缓存
├─ 根本方案：实现虚拟滚动、代码分割、懒加载
└─ 升华理解：性能优化是用户体验的第一道防线
```
