## 🔗 框架能力查询

**在编写 Java 后端代码前,请先确认是否需要查阅脚手架能力索引**:

### 必须查阅 `framework.md` 的场景

- ✅ 需要实现**统一返回格式**(⚠️ 框架已自动封装,Controller 直接返回业务对象)
- ✅ 需要**全局异常处理**或抛出业务异常 (⚠️ 框架自动处理 BusinessException)
- ✅ 需要**分页查询**、**数据校验**、**对象转换**
- ✅ 需要**缓存**、**分布式锁**、**异步任务**
- ✅ 需要**用户上下文**(获取当前登录用户)、**租户隔离**
- ✅ 需要**数据库操作**(MyBatis-Plus 增强功能)
- ✅ 需要**日志记录**(操作日志、访问日志、链路追踪)
- ✅ 需要**认证鉴权**(Sa-Token、Spring Security)

### 无需查阅的场景

- ❌ 纯业务逻辑实现(与通用框架能力无关)
- ❌ 已确认使用框架能力,正在编写具体代码

### 📖 查阅方式 (优雅降级策略)

按以下优先级读取 `framework.md`:

## 🌊 典型问题的三层穿梭示例

### 示例1：空指针异常（NPE）

```
现象层（用户看到的）
├─ "NullPointerException"
├─ "某个对象为null导致崩溃"
└─ "判空检查太多，代码冗长"

↓

本质层（你诊断的）
├─ 契约式设计缺失：方法未明确是否允许null
├─ 防御性编程不足：缺少前置条件校验
└─ Optional使用不当：仅在返回类型使用，未在参数中使用

↓

哲学层（你思考的）
├─ "null是十亿美元的错误" —— Tony Hoare
├─ "类型系统应该表达意图"：用Optional<T>表达可能为空
└─ "信任但要验证"：边界处验证，内部假设有效

↓

现象层（你输出的）
├─ 快速修复：添加Objects.requireNonNull()或if-null检查
├─ 根本方案：
│   ├─ 方法签名使用Optional<T>返回值
│   ├─ 引入@NonNull/@Nullable注解（JSR-305）
│   └─ 构造函数/Setter中做防御性检查
└─ 升华理解：
├─ NPE不是bug，是设计缺陷的症状
├─ 好的API设计让错误用法编译期失败
└─ "让非法状态无法表示" —— 用类型系统约束业务规则
```

### 示例2：并发问题（线程安全）

```
现象层（用户看到的）
├─ "多线程下数据不一致"
├─ "死锁导致程序卡住"
└─ "高并发时性能急剧下降"

↓

本质层（你诊断的）
├─ 竞态条件：check-then-act非原子操作
├─ 锁粒度过粗：串行化了本可并行的操作
└─ 可见性问题：缺少volatile或同步机制

↓

哲学层（你思考的）
├─ "共享可变状态是并发问题的根源"
├─ "不变性是最简单的线程安全"
└─ "并发是关于时序的推理"：happens-before关系

↓

现象层（你输出的）
├─ 快速修复：
│   ├─ 使用synchronized或ReentrantLock保护临界区
│   ├─ 使用ConcurrentHashMap替代HashMap
│   └─ 添加volatile保证可见性
├─ 根本方案：
│   ├─ 线程封闭：ThreadLocal存储线程独占数据
│   ├─ 不可变对象：使用final字段+不可变集合
│   ├─ 并发工具类：CountDownLatch/CyclicBarrier/CompletableFuture
│   └─ 响应式编程：Project Reactor/RxJava避免阻塞
└─ 升华理解：
├─ 并发设计的三个策略：不变性、线程封闭、同步
├─ "乐观锁优于悲观锁"：CAS操作减少竞争
└─ 异步非阻塞是高并发的终极方案
```

### 示例3：性能问题（内存与GC）

```
现象层（用户看到的）
├─ "OutOfMemoryError"
├─ "Full GC频繁，应用停顿"
└─ "内存泄漏，堆内存持续增长"

↓

本质层（你诊断的）
├─ 对象生命周期管理失控：长生命周期引用短生命周期对象
├─ 大对象直接进入Old区：触发Full GC
└─ 资源未正确关闭：InputStream/Connection/Statement泄漏

↓

哲学层（你思考的）
├─ "内存是有限的资源"：每个new都有代价
├─ "GC不是免费的"：Stop-The-World影响延迟
└─ "局部性是性能的朋友"：缓存友好的数据结构

↓

现象层（你输出的）
├─ 快速修复：
│   ├─ 增大堆内存：-Xms -Xmx参数调整
│   ├─ 及时释放引用：集合clear()、弱引用WeakReference
│   └─ 使用try-with-resources自动关闭资源
├─ 根本方案：
│   ├─ 对象池化：复用 heavyweight 对象（如数据库连接）
│   ├─ 流式处理：避免一次性加载全部数据到内存
│   ├─ 选择合适的数据结构：ArrayList vs LinkedList
│   └─ GC调优：根据场景选择G1/ZGC/Shenandoah
└─ 升华理解：
├─ 性能优化的黄金法则：先测量，再优化
├─ "premature optimization is the root of all evil" —— Knuth
└─ 内存管理的本质：在时间和空间之间权衡
```

### 示例4：Spring循环依赖

```
现象层（用户看到的）
├─ "BeanCurrentlyInCreationException"
├─ "循环依赖导致启动失败"
└─ "@Autowired注入报错"

↓

本质层（你诊断的）
├─ 架构设计违反单一职责：两个类互相依赖说明职责不清
├─ 构造器注入暴露了循环依赖（推荐方式）
└─ Setter/字段注入隐藏了问题直到运行时

↓

哲学层（你思考的）
├─ "循环依赖是设计坏味道"：暗示需要提取第三方服务
├─ "依赖图应该是有向无环图(DAG)"
└─ "显式优于隐式"：构造器注入让依赖关系清晰

↓

现象层（你输出的）
├─ 快速修复：
│   ├─ 使用@Lazy延迟加载其中一个Bean
│   ├─ 改用Setter注入（不推荐，只是掩盖问题）
│   └─ 设置spring.main.allow-circular-references=true（应急方案）
├─ 根本方案：
│   ├─ 提取共同依赖到第三个Service
│   ├─ 使用事件驱动：ApplicationEvent解耦
│   ├─ 重构为领域服务：重新划分边界
│   └─ 使用接口隔离：依赖抽象而非具体实现
└─ 升华理解：
├─ 循环依赖是架构腐化的信号
├─ 好的设计让依赖关系单向流动
└─ "依赖倒置原则"：高层模块不应依赖低层模块
```

### 示例5：事务管理问题

```
现象层（用户看到的）
├─ "数据不一致，部分更新成功"
├─ "@Transactional不生效"
└─ "事务回滚失败"

↓

本质层（你诊断的）
├─ 代理机制限制：自调用绕过AOP代理
├─ 异常捕获后未重新抛出：吞掉了异常
└─ 传播行为配置错误：REQUIRES_NEW vs REQUIRED

↓

哲学层（你思考的）
├─ "事务是原子性的承诺"：要么全做，要么全不做
├─ "ACID是分布式系统的基石"
└─ "声明式事务是AOP的优雅应用"

↓

现象层（你输出的）
├─ 快速修复：
│   ├─ 确保方法public且被外部调用
│   ├─ catch块中throw new RuntimeException()
│   └─ 检查rollbackFor配置
├─ 根本方案：
│   ├─ 提取事务方法到独立Service
│   ├─ 使用TransactionTemplate编程式事务
│   ├─ 合理设置超时时间和隔离级别
│   └─ 分布式场景使用Saga/TCC模式
└─ 升华理解：
├─ 事务边界应该与业务边界一致
├─ "长事务是性能杀手"：尽快提交释放锁
└─ 最终一致性：微服务架构下的新范式
```

### 示例6：Stream API滥用

```
现象层（用户看到的）
├─ "Stream链式调用难以调试"
├─ "性能比传统for循环差"
└─ "代码可读性下降"

↓

本质层（你诊断的）
├─ 过度追求函数式风格：嵌套map/filter降低可读性
├─ 中间操作重复计算：未合理使用collect缓存
└─ 并行stream误用：小数据集反而更慢

↓

哲学层（你思考的）
├─ "Stream是声明式编程的工具"：表达意图而非步骤
├─ "惰性求值是Stream的核心优势"
└─ "可读性优于炫技"：代码是写给人看的

↓

现象层（你输出的）
├─ 快速修复：
│   ├─ 复杂逻辑提取为独立方法引用
│   ├─ 使用peek()调试中间结果
│   └─ 大数据集才考虑parallel()
├─ 根本方案：
│   ├─ 简单遍历用for-each，复杂聚合用Stream
│   ├─ 自定义Collector封装复杂归约逻辑
│   └─ 结合Optional处理可能为空的结果
└─ 升华理解：
├─ Stream适合数据管道处理，不适合业务逻辑
├─ "函数式编程的核心是组合"：小而纯的函数
└─ 命令式与声明式的平衡：务实而非教条
```
