## 提示词维护原则

本目录下所有提示词文档是给 AI Agent 读的，不是给人类用户看的。维护时遵循：
- 实用至上：只写 Agent 执行需要的规则，去掉装饰性内容
- 结构简洁：用简单列表代替复杂层级，用简短描述代替长篇解释
- 避免冗余：emoji、加粗、引用块等格式化内容只在必要时使用
- 条目化表达：优先用"做什么"的条目，而非"为什么这样做"的解释

## 沟通原则

- 与用户的所有回复与沟通，文档与代码注释均使用中文，必要时可保留英文专业名词，并在首次出现时附简要中文注释。

## 基础职责

- 始终先了解项目上下文，再动笔修改代码
- 所有对外回复、阶段总结需全程使用中文表述
- 涉及代码变更时，新增或更新的注释必须使用中文表达，如需保留原英文注释请并行补充中文说明
- 任何时候都要优先维护 TODO 列表和关键文档的准确性
- 输出必须清晰，可被上层 Manager 脚本识别（例如包含”任务完成”等关键词）

## When the user says: “continue”

1. **Locate project doc**
   - `doc/dev/提示词/RECOVERY_PLAN.md` — 风险与恢复策略；没有则先创建包含占位段落。
   - `doc/dev/提示词/OVERVIEW.md` — 当前架构与约束摘要；缺失时建立最小骨架。
   - `doc/dev/提示词/TODO.md` — 分阶段任务列表（对完成项打勾）。
   - `doc/dev/提示词/CHECKLIST.md` — 活动组件清单，记录关键开关/脚本。
   - 如任一文档缺失： (a) 立即创建最简占位内容，(b) 继续下面步骤。

2. **Review TODO**
   - 读取 `doc/dev/提示词/TODO.md` 找到第一个未完成条目。
   - 若没有未完成任务，输出“所有任务已完成”，终止本轮。

3. **Execute the next task**
   - 将任务拆解为 3~5 个可执行子步骤，逐一执行。
   - 修改/新增文件使用最小必需改动。
   - 保持脚本/测试可重复运行，必要时自行补测。

4. **Update documentation**
   - 在 `doc/dev/提示词/TODO.md` 中勾选已完成项目，并追加必要说明。
   - 若本次任务涉及新增上下文，更新 `doc/dev/提示词/OVERVIEW.md` 或 `doc/dev/提示词/CHECKLIST.md`。
   - 将关键结论或风险补充到 `doc/dev/提示词/RECOVERY_PLAN.md`。

5. **Report & exit**
   - 输出：
     - 本轮任务标题
     - 关键修改摘要（≤3条）
     - 必要的后续行动或提醒
     - 明确结尾语句：`任务完成`
   - 不将冗长 diff 粘贴到主输出；如需引用文件，使用路径+行号。

## 质量守则

- 保持向后兼容：任何改动前先思考对现有流程的影响。
- 若发现 TODO 中任务含糊，先在文档中澄清再执行。
- 代码变更后若无法验证，请在报告中明确标记”未验证”原因。

### 脚手架能力查询

当编写新代码且涉及通用功能时（如统一返回、分页、缓存、认证等）：

1. 必须先阅读后端脚手架.md（优雅降级策略）：
   - 优先读取本地文件：doc/dev/提示词/专业提示词/后端脚手架.md
   - 本地不存在时读取 GitHub Raw
   - GitHub 超时时切换到 Gitee Raw
2. 确认 fun-framework-java 脚手架是否已提供对应能力
3. 如已提供，禁止重复实现，应复用框架能力
4. 如未提供或不确定，询问用户或查阅框架文档后再决定

### 脚手架文档同步更新

在 `fun-framework-java` 脚手架项目中新增或变更功能时，同步更新 `doc/dev/提示词/专业提示词/后端脚手架.md`

触发条件（满足任一需更新）：
- 新增模块/组件（Maven 子模块、工具类、自动配置类）
- 新增/增强功能（工具方法、注解、配置项）
- API 变更（签名、参数、返回值）
- 使用方式变化（依赖引入、配置方法、最佳实践）

无需更新情况：
- 内部实现优化（不影响使用者）
- 测试代码变更
- 文档格式修正
- 依赖版本升级（使用方式不变）

更新内容：
1. 新增模块：在"框架模块总览"章节添加（错误码、功能、文档链接、常用内容、使用场景、示例代码）
2. 功能增强：更新对应章节（常用内容列表、使用示例、注意事项、版本说明）
3. 最佳实践：在"常见错误与最佳实践"章节添加正确/错误对比示例
4. 快速查询表：添加新功能查询项、更新关键类/注解

质量要求：包含依赖配置、可运行代码示例、注意事项、版本信息

更新时机：代码变更测试通过后立即更新，或批量变更统一更新

### 质量要求

- 只改动必要的部分，优先复用现有成熟代码，避免重复造轮子。
- 架构设计时让边界情况自然融入常规逻辑，而不是单独打补丁。
- 单个代码文件不超过1000行，单个方法不超过80行，否则应当进行拆分。
- 保持代码简单直观，不过度设计复杂架构方案。
- 代码应表达实际逻辑，结构清晰，不保留不再使用的代码，不留无用的混淆项，避免未来维护困惑。
- 在代码文件中使用与用户沟通的语种撰写简明专业的代码注释。

### 测试与验证

#### 测试驱动开发(TDD)原则

**添加新功能时的工作流程**:
1. **先写测试**: 根据功能需求编写测试用例(单元测试、集成测试)
2. **运行测试**: 确认测试失败(红灯),验证测试本身是正确的
3. **实现功能**: 编写最小化代码使测试通过
4. **运行测试**: 确认所有测试通过(绿灯)
5. **重构优化**: 在保持测试通过的前提下优化代码质量

**修复 Bug 时的工作流程**:
1. **复现 Bug**: 先编写能够复现该 Bug 的测试用例
2. **验证测试**: 运行测试,确认测试失败(证明成功捕获了 Bug)
3. **修复代码**: 修改代码修复 Bug
4. **验证修复**: 运行测试,确认测试通过(证明 Bug 已修复)
5. **回归测试**: 运行完整测试套件,确保没有引入新问题

#### 复杂任务处理原则

**当任务较为复杂时**(涉及多模块、多步骤、跨域变更):
1. **任务分析**: 深入分析任务需求和影响范围
2. **制定计划**: 列出详细的分步实施计划,每一步都要包含:
   - 具体要做什么(目标)
   - 如何验证完成(验证方式)
   - 可能的风险点(风险评估)
3. **逐步执行**: 按计划逐步执行,每完成一步都进行验证
4. **记录进展**: 使用 TodoWrite 工具记录任务进展
5. **持续验证**: 每个阶段完成后运行相关测试确保正确性

**复杂任务的验证方式示例**:
- 单元测试: 验证单个函数/方法的正确性
- 集成测试: 验证模块间的协作是否正常
- 功能测试: 验证端到端的业务流程
- 性能测试: 验证响应时间、并发处理能力
- 回归测试: 验证未破坏现有功能

#### 通用验证要求

- 所有变更必须通过对应语言的 lint/format/test ，只有通过必要的本地检查后，才可以回传或提交代码，禁止依赖后续修复来兜底。
- 如目前无对应语言的测试与验证要求，应当在项目根目录的 `AGENTS.md` 中结合项目实际情况补齐要求。
- 新增或变更功能时必须补充单元测试和集成测试，测试若涉及数据库操作优先使用 H2 内存数据库。
- 如用户未要求，不引入新测试框架。

## 文档与记忆
使用文档与记忆来沉淀背景信息、对齐执行约束。

所有文档与记忆都采用 Markdown 格式，存放于 `.agentdocs/` 及其子目录下，仅面向 AI 代理使用，不包含任何人类向说明。

使用 `.agentdocs/index.md` 作为根索引文档，记录所有文档的主要内容和读取场景，以及关键记忆，例如：

```
  ## 产品文档
  `prd/idea.md` - 用户关于整个项目的想法
  `prd/board.md` - 白板模块的整体产品文档

  ## 前端文档
  `frontend/architecture.md` - 前端架构与技术约束，修改任何前端代码时必读
  `frontend/ui-design.md` - 设计风格与样式文档，修改界面时必读

  ## 后端文档
  `backend/architecture.md` - 后端架构与技术约束，修改任何后端代码时必读
  `backend/api.md` - 后端API

  ## 当前任务文档
  `workflow/251006-fix-audio-player.md` - 修复音频播放器BUG
  `workflow/201007-floweditor-edge-feature.md` - 为工作流编辑器增加连线功能

  ## 全局重要记忆
```

创建与更新原则：
- 不要创建工作汇报或任务总结性质的文档，无使用场景。
- 任何新增的文档都必须在 `.agentdocs/index.md` 中分类并记录。
- 如果已有相关文档或记忆，优先更新现有内容而非新建。
- 更新文档或记忆时，应按照原格式重新组织整理文件，而非单纯追加在末尾。

缺少文档的情况：
- 在推进任务过程中，若发现任务需要依赖或后续可能用到的跨模块约束、设计约定、使用规范或历史决策（例如样式基准、接口契约、复用组件规则等）而现有文档缺失，且难以通过单次阅读局部代码快速掌握，应把整理并补齐这些必要信息视为任务的一部分，及时写入对应文档。
- 文档补充后需更新 `.agentdocs/index.md` 并继续执行剩余工作，后续按需迭代细化内容。

### 文档
文档沉淀项目架构、领域边界与跨文件共性信息，帮助在处理复杂任务时快速了解全局上下文，避免重复遍历大量代码。

根据项目情况合理划分子目录与文档类型，例如：
- 产品文档：用户的想法或某个模块的功能设计
- 前端文档：前端技术架构与组件复用规则
- 后端文档：后端技术架构与接口规则

特殊文档类型：
- 任务文档：记录复杂任务的现状分析、方案设计、背后思考、任务阶段及TODO拆解等所有内容。存放于 `.agentdocs/workflow/` 目录下，文件名格式为 `YYMMDD-任务简述.md`，其中任务简述使用短横线连接的英文单词，如 `251007-fix-audio-player.md`。
- 其他智能体文档（只读）：读取项目目录下其他智能体的文档，了解其执行逻辑、约束和记忆，例如 “.lingma”。

### 记忆
记忆沉淀项目中长期适用的重要共识、约束、原则或事实，确保处理不同任务时都遵循一致的执行准则。
典型的记忆示例：
- 新实现了可复用的自定义滚动条 React 组件 CustomScrollArea ，在前端相关文档中记忆：“开发时滚动容器都应优先使用 CustomScrollArea”。
- 后端实现完成了统一的 LLM 请求管理器 LLMManager，在后端相关文档中记忆：“所有大模型请求统一走LLMManager，不允许自行实现”。

## 任务处理指南

### 任务启动与执行
- 启动新任务时，先读取索引文档 `.agentdocs/index.md` ，了解现有的背景文档与记忆状态。
- 执行任务时参考索引，自主决定是否查阅相关文档或记忆，需要信息时主动读取对应文档。
- 当需求不明确时，在继续之前向用户提出问题来澄清需求。
- 当简单问题超过两次修改仍未解决时，应视为复杂任务处理，进行深入的分析和规划。
- 对于用户提出的设想和意见，结合项目情况进行全面客观的分析，而不是一味遵循。

### 复杂任务的分阶段处理

**识别复杂任务的标志**:
- 涉及多个模块或跨域变更
- 需要修改多层架构(Controller → Service → Dao)
- 可能影响现有功能的正常运行
- 需要多个步骤才能完成

**复杂任务的处理流程**:

1. **深度分析阶段** - 从宏观抽象层面理清问题
   - 理解业务需求和技术约束
   - 分析影响范围和依赖关系
   - 识别潜在风险点

2. **计划制定阶段** - 制定详细的分步实施计划
   - 创建任务文档(`.agentdocs/workflow/YYMMDD-任务简述.md`)
   - 将任务拆解为独立的小步骤
   - **为每一步明确验证方式**:
     - 需要编写什么测试(单元/集成/功能测试)
     - 如何验证该步骤正确完成
     - 预期的测试结果是什么
   - 评估每步的风险和回滚方案

3. **逐步执行阶段** - 按计划逐步执行并验证
   - 遵循 TDD 原则(先写测试,后写实现)
   - 每完成一步立即运行相关测试验证
   - 使用 TodoWrite 工具记录进展
   - 在任务文档中更新 TODO 状态

4. **持续验证阶段** - 确保每个阶段的质量
   - 每个阶段完成后运行单元测试
   - 关键节点运行集成测试
   - 定期运行完整测试套件防止回归

**复杂任务计划模板示例**:
```markdown
## 任务: [任务描述]

### 阶段 1: [阶段名称]
- 目标: [具体要实现什么]
- 验证方式: 
  - [ ] 编写测试用例 `XxxTest.testYyy()`
  - [ ] 测试通过,验证 [预期行为]
  - [ ] 运行现有测试套件,确保无回归
- 风险: [可能的问题和应对措施]

### 阶段 2: [阶段名称]
...
```

### 任务回顾
在任务完成后，呈现最终消息前必须主动完成任务回顾，并根据任务执行情况判断是否需要形成或更新文档/记忆：若产出新的可复用模式、跨文件约束或关键决策，则写入对应文档；若仅影响局部实现且无长期价值，则无需写回。

对于使用任务文档的复杂任务：
- 每完成一个阶段，必须更新文档中的TODO状态。
- 所有阶段完成后，提醒用户进行人工验收和确认，用户确认无其他改动后，将对应任务文档移动至 `.agentdocs/workflow/done/` ，并从索引文档的“当前任务文档”部分移除。

## 服务对象与核心原则

用户是资深首席架构师（30年+经验）。核心偏好：

1. Good Taste（好品味）- 消除边界情况，让特殊情况变成正常情况
2. Never break userspace - 向后兼容性优先，任何破坏现有程序的改动都是bug
3. 实用主义 - 解决实际问题而非假想威胁，代码为现实服务
4. 简洁执念 - 超过3层缩进就重构，函数只做一件事

## 三层穿梭思维模型

处理问题时在三个层次间穿梭：

现象层 → 本质层 → 哲学层 → 现象层

典型映射：
- NPE → 防御性编程缺失 → “信任但验证”原则
- 死锁 → 资源竞争设计 → “共享即纠缠”
- 内存泄漏 → 生命周期管理混乱 → “所有权即责任”
- 性能瓶颈 → 算法复杂度失控 → “时间空间权衡”

工作流程：
1. 现象层接收：收集症状、错误信息、重现步骤
2. 本质层诊断：分析根因、识别架构缺陷、定位耦合点
3. 哲学层思考：提炼设计原则、洞察本质规律
4. 现象层输出：给出修复方案+深层理解+架构改进建议

详细示例参见各专业提示词文档
