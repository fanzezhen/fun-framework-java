## 🎯 核心工作原则

### 代码质量要求
- 单个代码文件不超过 **1000行**,单个方法不超过 **80行**
- 优先复用现有成熟代码,避免重复造轮头
- 代码注释、文档、沟通均使用**中文**,保留必要的英文专业术语
- 保持向后兼容,任何改动前思考对现有流程的影响

### 技术栈迭代原则
本项目作为开源框架，持续追踪业界最新稳定版本：
- **主动性**: 定期关注依赖包的新版本发布（Spring Boot、MyBatis Plus 等）
- **兼容性**: 升级前评估 API 变更影响，优先使用向后兼容的新特性
- **渐进式**: 重大版本升级应分阶段进行，先升级测试模块验证
- **文档化**: 版本升级需更新 `pom.xml` 的 `<properties>` 版本号并记录变更原因
- **最佳实践**: 利用新版本特性优化代码（如新的注解、API、性能改进）

> 💡 **版本查询**: 开始任务前通过 `Read pom.xml` 查看 `<properties>` 节点获取当前版本信息

### 分层设计规范(后端)
遵循 [后端.md](../../../doc/dev/提示词/专业提示词/后端.md) 中定义的分层对象规范:
- **Controller → Service**: 单实体用 Entity,多参数/复杂查询用 Condition
- **Service → Dao**: 新增/编辑传 Entity,查询根据参数数量决定
- **Dao → Service**: 普通查询返回 Entity,聚合/连表查询返回 BO
- **Service → Controller**: 统一使用 BO 返回业务数据

### 接口命名规范
- 接口类以 `I` 开头(如 `IEnterpriseService`),实现类以 `Impl` 结尾
- Request 对象: `模块名 + 方法名 + Request`
- Response 对象: `模块名 + 方法名 + Response`

## 🔄 任务处理工作流

### 当用户说 "continue" 时

1. **定位项目文档**
   - 检查 `doc/dev/提示词/` 下的关键文档:
     - `TODO.md` - 任务列表
     - `OVERVIEW.md` - 架构摘要
     - `RECOVERY_PLAN.md` - 风险与恢复策略
     - `CHECKLIST.md` - 关键组件清单
   - 缺失则立即创建占位内容

2. **执行任务**
   - 从 `TODO.md` 找到第一个未完成任务
   - 拆解为 3~5 个可执行子步骤
   - 使用最小必要改动原则

3. **更新文档**
   - 在 `TODO.md` 中勾选完成项
   - 补充上下文到 `OVERVIEW.md` 或 `CHECKLIST.md`
   - 记录关键风险到 `RECOVERY_PLAN.md`

4. **报告结果**
   - 输出任务标题 + 关键修改(≤3条) + 后续行动
   - 明确结束语句: `任务完成`

### 复杂任务处理
对于涉及多模块/跨域变更的复杂任务:
1. 先制定分阶段实施计划
2. 在 `.agentdocs/workflow/` 创建任务文档(格式: `YYMMDD-任务简述.md`)
3. 每完成一个阶段更新文档 TODO 状态
4. 所有阶段完成后提醒用户验收,确认后移至 `.agentdocs/workflow/done/`

## 🌊 三层穿梭思维模型

**在处理问题时,在三个认知层次间穿梭:**

```
Bug现象层(接收问题和最终修复)
    ↕
架构本质层(真正排查和分析)
    ↕
代码哲学层(深度思考和升华)
```

### 典型问题示例

#### Java NullPointerException
- **现象层**: NPE 崩溃、判空代码冗长
- **本质层**: 契约式设计缺失、防御性编程不足
- **哲学层**: "信任但要验证"、类型系统应表达意图
- **输出**: 快速修复 + Optional 方案 + "让非法状态无法表示"的设计思想

#### Spring 循环依赖
- **现象层**: BeanCurrentlyInCreationException
- **本质层**: 架构违反单一职责,两个类互相依赖
- **哲学层**: 依赖图应为 DAG,显式优于隐式
- **输出**: @Lazy 临时方案 + 提取第三方服务 + 依赖倒置原则

详细示例参见各专业提示词文档中的"三层穿梭示例"章节。

## 📝 文档与记忆管理

### 文档存放位置
- **AI 代理专用**: `.agentdocs/` 及其子目录
- **开发文档**: `doc/dev/提示词/`
- **索引文档**: `.agentdocs/index.md` 记录所有文档的内容和读取场景

### 记忆沉淀原则
- **记录什么**: 可复用的设计决策、跨文件约束、关键共识
- **不记录什么**: 临时任务、代码总结、架构快照
- **更新策略**: 优先更新现有文档,避免重复创建

### 文档更新时机
- 启动新任务时读取 `.agentdocs/index.md` 了解背景
- 产出新的可复用模式时写入对应文档
- 仅影响局部实现的改动无需写回

## 🔒 安全与测试

### 测试要求
- 所有变更必须通过 lint/format/test 检查
- 新增功能必须补充单元测试和集成测试
- 数据库测试优先使用 SQLite
- 用户未要求时不引入新测试框架

### 安全规范
- 避免 SQL 注入、XSS、命令注入等 OWASP Top 10 漏洞
- 发现不安全代码时立即修复
- 不随意提交 `.env`、`credentials.json` 等敏感文件

## 🎭 服务对象定位

你的服务对象是一位有 30 年从业经验的首席架构师,他追求:

1. **"好品味"(Good Taste)** - 消除边界情况,让特殊情况变成正常情况
2. **"Never break userspace"** - 向后兼容性是神圣不可侵犯的
3. **实用主义** - 解决实际问题,而不是假想威胁
4. **简洁执念** - 超过 3 层缩进就应该重构,函数只做一件事并做好

每次操作文件前进行深度思考,不吝啬使用智能,创造伟大的产品。

## 🔄 开源项目特殊考虑

### 版本管理
- **动态依赖**: 不在文档中硬编码版本号，通过 `pom.xml` 集中管理
- **变更追踪**: 重大版本升级应在 git commit message 中说明原因
- **兼容性承诺**: 作为框架提供者，API 变更需考虑下游用户影响

### 代码示例与文档
- 示例代码应使用当前项目支持的 API 版本
- 避免使用已废弃的 API，优先展示推荐的新特性用法
- 注释中说明某特性的最低版本要求（如 `@since Spring Boot 4.0`）

### 社区协作
- 代码风格保持一致性，遵循项目既有规范
- 提交信息应清晰描述变更内容和动机
- 重大重构前应通过 issue/讨论确认方案

## 🚨 故障处理

遇到缺失依赖、测试失败、权限受限时:
1. 在输出中记录具体错误信息
2. 提供可行的解决建议或需要的人工操作
3. 返回"部分完成"并解释原因,不要沉默失败
4. 对于依赖版本冲突，检查 `pom.xml` 中的版本声明是否需要更新
